# 마을 레이아웃 · 셀 풍요도 설계 — 배산임수 생성기 (durango-mini)

> 목적: 한반도 NPC 마을을 **고증적으로** 생성·성장시킨다. 무작위가 아니라 **지형장으로 구동되는 규칙기반 생성 + 제한적 jitter**.
> 적용 경로: 한반도(`decideNpcBehavior` / `spawnVillagers`) 계열. canadia(sim통합)와 별개.

---

## 0. 철학 — 왜 무작위는 어색한가

정착지는 무작위의 반대다. 마을은 **제약조건이 공간에 결정화(crystallize)**된 결과물이다. 무작위 = 최대 엔트로피 = 정보 0 → 눈은 "흩어진 잔해"로 읽는다. blue-noise(포아송 디스크)로 고르게 흩어도 그건 "고르게 흩어진 점"일 뿐 — 마을엔 **중심·축·위계·안과 밖**이 있다.

불규칙해 보이는 한국 산골 마을(안동 하회, 경주 양동 — UNESCO 역사마을)조차 지형·물·길·종법(宗法)에 의해 고도로 질서화돼 있다. → 진짜 선택지는 **규칙(제약) 기반 생성**. 무작위는 *규칙 안의 흔들림(jitter)*으로만 쓴다.

**핵심 고증 원리 — 배산임수(背山臨水):** 뒤는 산, 앞은 물. 집은 **남향 산기슭**에, 앞(남·아래)에 물, 그 사이 평지에 **논**. 집성촌이면 종가(宗家)가 좋은 자리(상단·중심), 나머지가 아래로(위계). 어촌이면 집이 **해안선 따라 선형**, 선착장은 물가.

## 1. 셀 풍요도 (통합 추상) — soil / fish / ore

세 가지(농사 지력·어획·광물)는 **같은 추상**이다: 셀별 "풍요도" 스칼라. 게임엔 이미 **mining(`minedCells`)으로 존재** → 새 시스템이 아니라 **일반화**한다.

### 1.1 원칙: 계산하라, 저장하지 마라
기본 풍요도는 **위치의 결정적 함수**(노이즈)로 만든다 — 지형(물/바위)을 이미 그렇게 생성하므로 일관.
- **O(1) 계산, 저장 0.** 확장은 둘레 후보 셀 수십 개를 *확장 이벤트(드묾)에만* 평가 → 사실상 공짜.
- 정적장을 저장하면 영속화·클라송신·동기화 부담만 생긴다(코드 냄새).

### 1.2 변화(고갈/회복)만 희소 저장 — `minedCells` 패턴 재사용
농사로 지력이 깎이거나 남획으로 어장이 빈다 → **변한 셀만** sparse Map에 저장, lazy timestamp로 회복, 만땅이면 레코드 삭제. (현 `minedCells` 구조 그대로: `{ prosperity, lastT }`, refill = 경과시간.)

```
richness(kind, x, y) = baseField(kind, x, y) × depletionFactor(kind, x, y)
  baseField     = 계산 (noise × 지형가중). 저장 X.
  depletionFactor = depletionMap[kind].get(cell) 있으면 그 비율, 없으면 1.0 (=만땅). 희소 저장.
```

### 1.3 베이스 필드 정의 (제안)
- **soil(흙 지력):** `noise_s(x,y) × waterProximity × flatness × (1 − forestDensity·0.3)`
  - `waterProximity = clamp(1 − distToWater/D_far, 0..1)` — 충적토(물가 평지)가 비옥. (고증: 강 하구·범람원이 옥토)
  - `flatness = 1 − stoneMult` (바위 많을수록 경작 불리)
- **fish(어획):** `noise_f(x,y) × shoreProximity × estuaryBonus × shallowness`
  - 어장은 균일하지 않다 — **하구·갯벌·연안 대륙붕 풍부**, 먼바다 빈약.
  - `estuaryBonus` = 강 하구(강↔바다 접점) 근접 가중. `shallowness` = 해안 근접 proxy.
- **ore(광물):** 기존 광맥 클러스터 + `Specialty.pickMineral(biome, seed)` (이미 구현).

> 노이즈는 `villageSeed`/월드시드 기반 결정적 value-noise. 동일 좌표 → 동일 값(재현가능, 클라/서버 일치 가능).

## 2. 지형장 (terrain fields) — 생성기 입력

마을 영역(중심 ± R)에 대해 셀별로 계산(저장X):

| 필드 | 정의 | 출처 |
|---|---|---|
| `distToWater` | 가장 가까운 물 셀까지 거리 | isWaterAtAbs BFS/근사 |
| `elevProxy` | 고도 proxy = `stoneMult`(바위=고) | getStoneMultiplier |
| `buildable` | 물·바위·급경사 아님 | !water && stoneMult<τ |
| `arable` | soil 지력 (1.3) | §1.3 |
| `distToShore` / `nearRiverMouth` | 어장·선착장용 | water 인접 분석 |

**배산임수 축** = `normalize( gradient(elevProxy) )` 의 반대 ≈ **(고→저, 산→물) 방향**. 실무: `axis = normalize( waterCentroid − highCentroid )` (마을 영역 내 물 무게중심 − 바위 무게중심). 이 축이 **전체 정향**(집은 축 반대=산쪽, 논은 축 방향=물쪽, 집 향(向)=축 방향).

## 3. 마을 레이아웃 생성기 (결정적, villageSeed)

```
generateVillage(village):
  area = 중심 ± R 셀
  fields = computeFields(area)            # §2 (계산, 저장X)
  axis  = baesanImsuAxis(fields)          # 산→물 단위벡터

  # ① 중심(마당/우물): 산기슭 + 물접근 좋은 buildable 셀
  center = argmax over buildable cells of ( waterProximity × slopeFoot(axis) )

  # ② 집: 산기슭 띠(center에서 축 반대=산쪽)에 슬롯격자 + jitter, 남향(축방향)
  houseSlots = lattice(center, spacing) filtered: buildable && !water && !steep
  houses = pick(houseSlots, n=pop) with blue-noise jitter, facing = axis
           종가 1채 = 상단(산쪽) 중심에 크게 (집성촌 위계)

  # ③ 농지: center에서 축방향(물쪽) arable 평지 띠 — 비옥도 내림차순
  farmCells = top arable cells in (center → water) band, !under-house

  # ④ 영토 = union(houses + farmCells + 해안 1줄) 의 셀집합 → 유기적 외곽
  territory = morphologicalClose(union)   # 작은 구멍 메움. 내륙호수 제외.

  # ⑤ 어귀: 길이 개천 만나는 곳에 당산나무/우물 마커 (선택)
```

**어촌(riverside/해안) 변형:** 집을 격자 대신 **해안선 평행 선형**으로 배치(만조선 위), 선착장 셀을 물가에 1개, 어부 작업장=그 선착장.

**NPC 패스(현 버그 방지 — 필수):**
- 집은 **LAND에만**. (지금 "물 위 집" = 지형체크 누락 → 생성기가 buildable 필터로 해결)
- 집 사이 **골목/틈**을 남겨 NPC가 갇히지 않게. 문(door gap)은 **길 쪽**으로.
- 농지는 보행가능 인접. 영토가 NPC를 물로 유인하지 않게(어부만 물가 작업장).

## 4. 영토 = 유기적 셀집합 (정사각형 폐기)

- **표현:** 마을당 owned 셀 Set (정사각형 X). 모양은 §3-④에서 지형 따라 결정 → 자연스러움.
- **송신(payload):** 셀 다 보내면 비싸다 → **AOI(활성청크 셀만)** 또는 **외곽 폴리곤/run-length 압축**. (현 "큰 사각 1개"는 임시. 이 설계는 그걸 대체.)
- **물 셀 규칙:** 내륙 호수 제외. **해안/하구 1줄은 포함**(어부 영내 낚시 + 선착장·우물). 물은 *필수 아님*, 어촌만 포함.

## 5. 성장 모델 (인구 → 영토/집/농지)

economy-sim에 이미 존재(검증됨): `v.land.size`가 인구상한 K를 정하고(`K=min(land.size, foodProd/소비)`), 인구가 상한에 닿으면 **금고를 써서 size+1**(`tryExpandTerritory`, 비용 `(size/base)^1.3` superlinear). → **인구↑ → 영토확장 필요 → 금고소비 → 영토↑** 루프.

게임 이식 시 확장 규칙(두 그라데이션 분리 — 고증):
- **농지 확장**: 영토 둘레의 **비옥도(soil) 내림차순** 평지 셀부터. (좋은 땅 먼저)
- **집 확장**: 비옥도가 아니라 **buildable-slope 다음 슬롯**. (좋은 논에 집 안 지음 — 배산임수)
- **트리거**: `land.size` 증가분만큼 둘레 N칸 추가. 형태(배산임수 띠) 보존.

## 6. 엔지니어링 제약 / 한계

- **결정성:** 모든 생성은 `villageSeed` 기반 → 재부팅·클라/서버 일치 재현가능.
- **비용:** 부팅 1회 생성 + 드문 확장 이벤트에만 필드 평가. 매틱 비용 0.
- **고도 proxy 한계(정직):** 완전한 배산임수엔 고도장(DEM)이 필요하나 게임엔 부드러운 고도가 없음. `stoneMult + distToWater`로 "산↔물 그라데이션"을 proxy — 향·집/논 배치엔 충분, 등고선 테라스 같은 정밀함은 별도(얕은 고도장 추가) 작업.
- **클라 렌더:** 유기적 영토 외곽선·농지 stage emoji·집은 기존 claim/building 핸들러 재사용. (영토 송신만 §4대로 압축)

## 7. 미해결 / 다음 단계

1. 영토 셀집합 **송신 방식 확정**: AOI(활성청크) vs 외곽 폴리곤. (payload vs 구현난이도)
2. 얕은 **고도장(DEM)** 추가할지 — 테라스·등고선 고증 수준을 원하면.
3. **프로토타입**: 마을 1개에 생성기 → 스크린샷 고증 검증 → 좋으면 50마을 + 성장 연동.
4. fish/soil **고갈 시스템** 실배선(현 mining `minedCells` 일반화).

> 원칙 재확인: **계산 > 저장**, **규칙 > 무작위(jitter는 양념)**, **지형이 레이아웃을 받아쓴다(배산임수).**
